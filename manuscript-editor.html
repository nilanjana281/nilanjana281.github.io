AI Manuscript Editor Agent

Designing and building a multi-provider AI editorial agent from scratch — integrating Claude, ChatGPT, and Gemini — to deliver consistent, professional-grade feedback on long-form fiction at chapter scale.

Key Problem

Existing AI tools fail long-form fiction — structurally, not just stylistically

Existing AI writing tools provide generic, stateless feedback. They work reasonably well for a standalone blog post or a short piece — but they fall apart for long-form, serialized fiction. The problems are structural: feedback across chapters is disconnected because there is no memory of earlier character decisions, plot threads, or voice choices. There is no continuity mechanism.

Compounding that, no tool combined multi-provider AI routing with structured editorial output and document export in one place. Authors using these tools were copy-pasting chapters into generic chat interfaces, getting inconsistent feedback with no shared context, and manually managing the gap between "AI suggestion" and "chapter edit." The workflow was fragmented, the output was inconsistent, and there was no way to track what feedback had been applied across a 70+ chapter manuscript.

The gap was not a missing model. It was a missing product layer — the system design, the routing logic, the output structure, and the state management that would make AI feedback genuinely useful at scale.

Core User

One author. Seventy-three chapters. Consistent editorial feedback needed across all of them.

An author working on a 73-chapter serialized novel needed professional-grade editorial feedback on prose quality, structure, character voice, and pacing — at scale, consistently, without losing continuity across chapters. The feedback had to be comparable chapter-to-chapter (so the author could trust it), specific enough to act on, and exportable in a form that could be archived or shared.

The user need was not "better AI." It was a system where AI feedback was reliable, consistent across sessions, structured enough to be actionable, and organized enough to track — none of which any existing tool provided out of the box.

Action

Building the product layer — not just using the API

Multi-Provider Routing

Integrated Claude, ChatGPT, and Gemini APIs and designed routing logic to select the optimal model per task type — rather than treating all models as interchangeable. Each model has different strengths across different editorial dimensions: prose-level feedback, structural analysis, pacing, and voice consistency respond differently across providers. The routing layer was designed to exploit those differences systematically.

Claude

Prose-level feedback, nuanced voice and tone analysis, character consistency

ChatGPT

Structural analysis, plot coherence, chapter arc evaluation

Gemini

Pacing analysis, scene rhythm, dialogue flow evaluation

Structured Output Contracts

Engineered prompts to produce structured JSON output across all three providers — enforcing a consistent output schema regardless of which model handled the task. This was the non-obvious architectural decision: without defined output contracts, switching models produces incomparable results. You cannot evaluate whether feedback improved or regressed if the format changes between sessions.

The JSON schema defined categories (prose quality, structure, character voice, pacing), severity levels, specific line references, and recommended actions — making feedback not just readable but trackable and comparable across chapters.

Session State, Chapter Tracking, and Export

Built chapter tracking and session state management so the system maintained context within a session — knowing which chapters had been reviewed, what feedback had been flagged, and what was pending. Built PDF and Word export so the author could archive chapter feedback, share it, or maintain a running editorial record alongside the manuscript.

Defined a v2 product roadmap covering persistent cross-chapter memory (so feedback on chapter 45 could reference character decisions from chapter 12) and highlighted track-changes output — the natural next step from structured feedback to actionable edits.

The core PM insight:

This project was not about finding the best model. It was about designing a system where any model could be swapped without breaking the experience — because the output contract, the routing logic, and the state management lived in the product layer, not inside any single model's behavior. That is the difference between using AI and building with it.

Impact

Live on GitHub. Structured editorial feedback at chapter scale.

The agent is live on GitHub and delivers structured, professional editorial feedback across prose quality, structure, voice, and pacing — consistently, at chapter scale. Feedback is comparable chapter-to-chapter, exportable, and organized by category and severity.

Beyond the specific output: this project demonstrates hands-on 0-to-1 AI product development — not prompting an existing tool, but designing and building a product layer on top of multiple AI APIs. Multi-provider routing, structured output engineering, session state management, and a defined v2 roadmap are all product decisions, not engineering ones.

Technical

Multi-provider routing across Claude, ChatGPT, and Gemini APIs. Structured JSON output contracts. Chapter tracking and session state. PDF and Word export. Deployed to GitHub.

Product

0-to-1 scoping from user need to v1 definition. Routing logic as a product design decision. V2 roadmap defined: persistent cross-chapter memory and track-changes export.

Learning

What building this taught about AI product development

Multi-provider routing requires structured output contracts to work. Switching models without defining expected output shapes produces inconsistent, hard-to-compare results — which undermines the core value of having routing at all. The output schema is not a nice-to-have; it is the foundation that makes the routing decision meaningful.

The PM skill in AI product development is not knowing which model is "best." It is designing the system so any model can be swapped without breaking the experience. That means the product layer — the routing logic, the output contracts, the state management — needs to be defined before the model selection is finalized.

And the clearest learning: the gap in AI tools for long-form creative work is not model capability. It is product design. The models can already do the work. What is missing is the system that makes their output reliable, comparable, organized, and continuity-aware across a document that spans months and dozens of chapters.
