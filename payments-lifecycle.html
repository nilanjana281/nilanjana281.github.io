Case Study · Fintech · Payments

Payments Lifecycle Integration Prototype

Building a hands-on prototype to map, test, and communicate the full payment lifecycle — authorization through refund — with explicit frontend-to-backend state transitions at each stage.

Fintech API Testing Payments Integration Design

Key Problem

Payments aren't a single flow — and treating them as one creates problems

Payment integrations are rarely just about enabling checkout. They involve multiple distinct transaction states — authorization, capture, cancellation, and refund — each with its own frontend behavior, backend dependencies, and order state transitions. When the entire lifecycle is treated as one continuous linear flow, several things break down.

Testing becomes tangled: you cannot isolate behavior at authorization without inadvertently triggering capture logic. Stakeholder communication becomes vague: explaining "payment" as a single action obscures the conditional logic that determines what the system actually does at each stage. Edge case identification becomes reactive rather than proactive, because the mental model doesn't distinguish between states until something goes wrong in production.

The core question this prototype was designed to answer: what does each transaction state actually do — at the API level, and at the frontend layer — and how do those two layers connect?

Core Users

The people who need to understand this — not just the ones who build it

Product and engineering teams evaluating, integrating, or communicating how third-party payment solutions behave across the full order lifecycle — not just at the checkout initiation point.

PMs and Technical PMs

Need to write specs and acceptance criteria that map frontend user actions to backend transaction outcomes at each lifecycle stage — without leaving "payment behavior" as a black box in the PRD.

Engineering Teams

Need testable, independently verifiable documentation of each state — so integration work can be validated in stages, and edge cases around order state transitions can be identified before they reach production.

Action

Build it, test it, document it — stage by stage

Built a hands-on prototype using Postman for API testing and CodePen for frontend payment widget interaction. The key structural decision: rather than treating the payment journey as one continuous flow, intentionally separated the lifecycle into distinct order scenarios — one scenario per transaction state. Each stage was independently testable and explainable.

Stage 1

Authorization

Card validated, funds reserved but not captured. Frontend: payment form submission + pending state. API: auth request, response with authorization token and status.

Stage 2

Capture

Reserved funds moved to merchant account. Frontend: order confirmed state triggered. API: capture call referencing auth token, settlement response and timing behavior.

Stage 3

Cancellation

Auth voided before capture. Frontend: order cancelled state, no charge confirmation. API: void request, release of reserved funds, edge cases around timing window.

Stage 4

Refund

Post-capture return of funds. Frontend: refund initiated and confirmed state. API: refund request with amount, partial vs. full refund logic, settlement reversal behavior.

At each stage: validated request-response behavior at the API level via Postman, then connected frontend user actions to backend transaction outcomes using the CodePen widget. This made the prototype both technically verifiable and communicable to non-engineering stakeholders — the widget interaction told the user story, and the Postman collection showed what the system was actually doing underneath.

Documented each stage with explicit mapping: what the user sees, what the frontend sends, what the API returns, what the order state becomes, and what the edge cases are at that stage specifically.

Impact

Clear, testable documentation of the full payment lifecycle

Produced explicit frontend-to-backend mapping across all four transaction states. Each stage is independently testable, independently documentable, and independently communicable — which is not the case when the lifecycle is treated as a single integration unit.

Deepened practical fluency in payment state transitions, API request-response structures, and how to structure payment flows in a way that is operationally meaningful — not just technically functional. The prototype also surfaced non-obvious edge cases around timing windows (particularly the cancellation-after-capture boundary) that are easy to miss when thinking about payments at the flow level rather than the state level.

The artifact functions both as a reference for integration work and as a stakeholder communication tool — the widget interaction layer makes the lifecycle legible to product, ops, and commercial stakeholders who don't live in API documentation.

Learning

The real product challenge in payments isn't enabling checkout

Separating lifecycle stages into distinct scenarios makes payment behavior dramatically easier to validate, communicate, and troubleshoot — for both PMs and engineering teams. The structure that seems like extra overhead at the start is the structure that prevents production incidents at the end.

The real product challenge in payments is not enabling checkout — it is reasoning clearly about what happens after. Authorization is not payment. Capture is not confirmation. Cancellation has a timing window. Refund has partial vs. full logic. Every one of those distinctions matters operationally, and the PM who understands them is the one who writes specs engineering can actually build from.

And building the prototype rather than just reading the API docs made the difference. There are behaviors you only discover by firing the requests — the timing edge cases, the error states that don't match the documentation, the frontend race conditions that only appear when you connect both layers simultaneously.
